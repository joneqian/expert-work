线框里的设计决策(点名说同意/改)

  • 纵轴 = 分类型的执行轨迹,不只 agent 步:agent = 大卡(思考+工具/答复);memory_recall / planner / reflect / memory_writeback = 轻量类型行(带 node 标签,可展开看内容);compaction / retry / error / end = 轴上菱形/圆点标记;approval = 紫色暂停标记(run 停在哪等审批)。全部按真实序列穿在一条轴上。见底部图例。
  • agent 步卡:标题行 步骤号 · node · 模型 · finish_reason · token · 耗时;工具卡嵌在所属步里(不再全局扁平)。
  • reflect 的 revise 用告警色高亮 —— 它会回环让 agent 重做,是"为什么循环了这么多步"的关键线索。
  • 异常步默认展开 + 红节点 + 红条(失败/重试/撞长度);正常步默认折叠,一行摘要,点开看思考/工具/答复。
  • 思考(reasoning)按步内联 —— reasoning_content 挂在每条 AI message 上(每次 LLM 调用),放在该步"动作"之前。删掉了旧的轮级「思考过程」大坨。需要 summarizeTurn 像 per-step token 那样保留每步 reasoning 关联。
  • 过滤条:类型 chip(全部/工具/错误/retry)+ 文本搜索(工具名/error/finish_reason),右侧实时计数。过滤 = 只留命中步(非命中步隐藏)。
  • 工具卡显 入参(args)+ 出参(结果)两段(复用 Batch 1 的 ToolTimeline 折叠):入参 = 调用参数 JSON;出参 = 结果 —— exec_python 走 exit_code 徽章 + stdout/stderr 分区,其它工具走结果预览。成功/失败状态在卡头。
  • retry / 工具失败 以 chip 挂在对应步内(数据来自 Batch 2 的 parser)。
  • 「原始事件」模式不变(仍是 Batch 2 的共享 EventCard 列表),只是这个过滤条也作用于它。
  • 待定 · 与 Batch 2「执行状态」聚合区的关系:时间线现在按序列展示了 memory/plan/reflect 等,和 Batch 2 的聚合卡片重叠了。选项:(a) 时间线管序列、聚合区管"全局汇总"(如最终 plan、全部记忆)两者并存;(b) 时间线吃掉聚合区。倾向 a(plan 走 PlanPanel REST 是最终态,和逐帧序列天然分工)。
