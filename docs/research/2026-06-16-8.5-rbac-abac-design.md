# 8.5 细粒度 RBAC-ABAC — 设计（资源实例 + 属性条件）

> 日期：2026-06-16
> 目标：满分化 8.5 权限粒度 ★3→★5。现状=资源**类型**级 RBAC（`(role, resource_type, action)` 矩阵，
> deny-by-default）；gap（评估原文）=「缺资源 URI 级 / 细粒度 RBAC-ABAC」。用户拍板 **Option 2**：
> RoleBinding 加 `conditions`，统一覆盖 URI 级（resource_ids）+ ABAC 属性（labels / owner_only）。
> 纯逻辑确定性，无模型、无运维依赖；CI 可单测 + 集成测。

---

## 0. 现状（基于代码核实）

- `auth/rbac.py:is_allowed(principal, resource, action)`：`(Role, Resource, Action)` 矩阵，纯函数，
  deny-by-default。Role ∈ {ADMIN, OPERATOR, VIEWER, SYSTEM_ADMIN}。
- `api/_authz.py:require(resource, action)`：FastAPI 依赖，类型级门 + 拒绝审计。
- `RoleBinding`（protocol `service_account.py` + ORM `models/role_binding.py`）：subject→role，
  tenant/platform scope。**无资源选择器、无条件**。
- `auth/tenant_roles.py:resolve_tenant_roles`：AuthMiddleware 把 subject 的 tenant-scope binding 角色
  **无条件并入** `principal.roles` → `is_allowed` 读到。
- owner/labels 数据源已在：`AgentSpecRecord.created_by`（owner）、`AgentMetadata.labels`（属性）。
- admin-ui 已有 IAM 面：`SettingsRoleBindings.tsx` / `Members` / `role_bindings.ts` SDK。

→ 缺的是**实例/属性维度**：无法表达「X 只是 agents[foo] 的 operator」「只能动 team=支持 的 agent」「只能动自己建的」。

## 1. 模型

### 1.1 `BindingConditions`（protocol，frozen）

```python
class BindingConditions(BaseModel):
    model_config = ConfigDict(frozen=True, extra="forbid")
    resource_ids: tuple[str, ...] = ()      # URI 级：仅这些实例 id/name（空=任意）
    labels: Mapping[str, str] = {}           # ABAC：资源 metadata.labels 须 superset 匹配（空=任意）
    owner_only: bool = False                 # ABAC：仅 owner==subject 的资源
```

`RoleBinding.conditions: BindingConditions | None = None`。**None = 无条件**（类型级，今日行为）。
校验：`conditions` 仅 tenant-scope 合法；platform_scope（system_admin 跨租户）必须 `conditions=None`。

### 1.2 评估器（`auth/abac.py`，纯函数）

```python
@dataclass(frozen=True)
class ResourceAttrs:
    resource_id: str | None = None
    labels: Mapping[str, str] = field(default_factory=dict)
    owner_id: str | None = None

def conditions_match(conditions, *, attrs, subject_id) -> bool:
    # resource_ids 非空 → attrs.resource_id ∈ resource_ids
    # labels 非空 → 每个 (k,v) 须 attrs.labels[k]==v
    # owner_only → attrs.owner_id == subject_id
    # 三者 AND；全空 → True
```

### 1.3 决策语义（**additive / most-permissive，安全关键，显式记录**）

标准 RBAC+ABAC 加性模型：grant 取并集，最宽者胜；本期不做 explicit-deny（记 §6 后续）。

- **无条件授权**（JWT realm 角色 / system_admin / **无条件** binding）→ 类型级，对任意实例放行。
- **有条件 binding** → 仅对**条件匹配的实例**放行（实例级新增授权）。

**关键**：`resolve_tenant_roles` 改为**仅并入无条件 binding 的角色**到 `principal.roles`；
**有条件 binding 不并入**（否则其角色会变成类型级授权，绕过条件）。
→ 要把用户**限制**到细粒度，须**仅经有条件 binding 授权**（不给宽 realm 角色 / 无条件 binding）。

### 1.4 路由决策 `require_resource(resource, action, attrs_loader)`

```
1. is_allowed(principal, resource, action) → 放行            # 无条件快路径
2. 否则：取 principal 的有条件 tenant binding；
   任一 binding 的 role grants (resource, action) ∧ conditions_match(attrs_loader()) → 放行
3. 否则 403（拒绝审计同 require）
```

`attrs_loader` 按资源加载 `ResourceAttrs`（lazy，仅快路径未命中才查资源）。有条件 binding 经
`role_binding_store.list_for_subject(tenant)` 取（慢路径 1 次索引查，低频）。**未升级到 require_resource
的路由**：有条件-only 用户被类型门拒（安全默认）；条件只在采用实例级 authz 的资源生效。

## 4. 测试

- 单测：`conditions_match`（各条件 + 组合 + 全空）；`authorize_resource` 加性语义（无条件胜 / 有条件实例级）。
- 集成：建有条件 binding 的 user，打 agents 路由——匹配 200 / 不匹配 403；无条件 user 不受影响（回归）。
- admin-ui：conditions 编辑器 vitest + Storybook。
- 确定性，无模型 → CI 全覆盖（同 15.3/3.3 档，无需 live E2E）。

## 5. 安全核对

- deny-by-default 不变（类型矩阵仍是第一道门）。
- 有条件 binding 不并入 `principal.roles` → 不会意外放大成类型级（§1.3 关键修正）。
- 租户隔离不动（tenant_id scoping + RLS + allowed_tenants 仍在）；ABAC 只在租户内细分。
- conditions 仅 tenant-scope；platform system_admin 跨租户不受条件约束（跨租户管理 UX 不破）。

## 6. 不做 / 后续

- explicit-deny / 优先级覆盖（加性模型够 ★5；deny 规则后续）。
- 路由全量采用：本期 agents 作样板，sessions（已有 J.14 owner 检查）/secrets 后续按需采用。
- 策略语言（Cedar/OPA）：用户否决，过度工程。
- binding 解析 TTL 缓存：M1（resolve_tenant_roles 注释已标 per-request M0）。
